# Соглашения по разработке моделей в OpenSCAD (Model Development Conventions)

В данном документе зафиксированы соглашения по стилю кода, структуре файлов и лучшим практикам 3D-моделирования, принятые в репозитории `cads`. Эти правила помогают сохранять читаемость кода, обеспечивать гибкую параметризацию моделей и поддерживать высокую производительность при рендеринге и предпросмотре.

---

## 1. Общая структура файла (`.scad`)

Каждый файл модели должен иметь предсказуемую и последовательную структуру разделов:

1. **Библиотеки и зависимости**: Инструкции `include` и `use` (в самом начале файла).
2. **Параметры сглаживания кривых (`$fa`, `$fs`)**: Динамическая настройка точности.
3. **Параметры Customizer**: Переменные, доступные пользователю через графический интерфейс OpenSCAD.
4. **Константы и внутренние параметры**: Раздел `/* [Hidden] */` для скрытых расчётных величин.
5. **Вычисляемые (производные) переменные**: Автоматический расчёт зависимых размеров.
6. **Описания модулей (`module`)**: Изолированные компоненты модели.
7. **Генерация (инстанцирование) модели**: В самом конце файла (часто с возможностью переключения/комментирования отдельных деталей).

---

## 2. Подключение внешних файлов (`include` vs `use`)

В OpenSCAD критически важно разделять способы импорта файлов:
* Используйте `include <path>` только тогда, когда вам необходимо импортировать глобальные переменные, константы или выполнить верхнеуровневый код.
* Используйте `use <path>`, если вам нужны только определения модулей и функций без выполнения кода верхнего уровня, содержащегося в подключаемом файле (это предотвращает случайную генерацию нежелательных геометрических объектов).

**Пример:**
```openscad
include <BOSL2/std.scad>      // Подключение стандартной библиотеки BOSL2
use <./canvas_modules.scad>  // Импорт только функций и модулей
```

---

## 3. Настройка точности и разрешения кривых (`$fa`, `$fs`)

Для баланса между скоростью предпросмотра (Preview) и качеством финального рендеринга (Render) используется адаптивное переключение параметров `$fa` (минимальный угол сегмента) и `$fs` (минимальный размер сегмента) с помощью системной переменной `$preview`:

```openscad
$fa = $preview ? 12 : 1;
$fs = $preview ? 2 : 0.4;
```

* **При предпросмотре (`$preview == true`)**: Используются грубые настройки сегментации, что делает вращение и зуммирование модели в окне просмотра мгновенным.
* **При рендере (`$preview == false`)**: Углы скругляются с высокой точностью для качественной печати или экспорта в STL.

## 4. Соглашения по именованию (Naming Conventions)

* **Глобальные параметры (Customizer)**: Пишутся в формате **PascalCase** (например, `StartNum`, `Grid_Size_mm`, `Corner_Radius_mm`).
* **Описание параметров Customizer — комментарием на строку выше**: Описание/подсказка параметра, видимая в GUI OpenSCAD Customizer, должна располагаться **отдельной строкой-комментарием непосредственно над объявлением** параметра, а не в конце строки после `;`. Inline-комментарии в конце строки не подхватываются Customizer-ом как описание. **Исключение — варианты выпадающего списка**: синтаксис `// [opt1, opt2, ...]` должен оставаться **на той же строке** после `;` (это требуется парсером Customizer-а). Внутренние/расчётные переменные (не для Customizer) можно комментировать inline.

**Правильно (описание над параметром, выпадашка — на той же строке):**
```openscad
// Диаметр хвостовика лампы
Shank_D_mm = 10;
// Режим вывода
Mode = "assembly"; // [assembly, print, front, back]
```

**Неправильно (описание inline; выпадашка перенесена вверх):**
```openscad
Shank_D_mm = 10; // диаметр хвостовика лампы
// [assembly, print, front, back]
Mode = "assembly";
```

### 4.1 Customizer — что попадает в форму (требования парсера)

* **Только главный файл.** Customizer сканирует переменные **только в основном `.scad`-файле**. Переменные из `include`/`use` в форму **не попадают** (но остаются определены — см. ниже про переопределение). Поэтому все настраиваемые параметры объявляйте в главном файле модели.
* **Только до первого модуля.** Переменные должны идти **выше первой строки с объявлением модуля** (со составным телом); всё ниже первого `module` в форму уже не попадает. Чтобы явно «обрезать» сканирование посреди файла, используют пустой ограничитель:
  ```openscad
  module __CustomizerLimit__ () {}   // всё ниже скрыто из Customizer
  ```
* **Только литералы.** Значение параметра обязано быть **буквальным** — число, строка, булев, вектор литералов. Выражения (`f = 12 + 0.5;`, `Slab = 3 * LineW;`, `str(...)`) параметрами **не являются** и в GUI не появятся. Отсюда правило: **все производные/расчётные величины прячьте в секцию `/* [Hidden] */`** — там выражения разрешены и не показываются как виджеты.
* **Переопределение библиотечной константы.** Если константа объявлена в подключаемой библиотеке (`include`), а её нужно сделать настраиваемой — повторно объявите её **литералом в главном файле выше точки использования**. В OpenSCAD выигрывает последнее присваивание, поэтому главное определение перекрывает библиотечное и попадает в Customizer:
  ```openscad
  include <./lib.scad>     // в библиотеке: Min_Hole_mm = 1.0;
  // Минимальный размер отверстия, мм
  Min_Hole_mm = 1.0;       // теперь это параметр Customizer
  ```

### 4.2 Customizer — типы виджетов и синтаксис

Описание — комментарием **строкой выше**; спецификатор виджета — **на той же строке** после `;`. Диапазон/шаг задавайте явно — Customizer угадывает их неточно (не знает про дробные/отрицательные значения) и полезно ограничивать рамки разумными для модели.

```openscad
// Слайдер с диапазоном
Plain_slider = 34;          // [10:100]
// Слайдер с шагом [start:step:stop]
Step_slider   = 2;          // [0:5:100]
// Выпадающий список (числа)
Numbers       = 2;          // [0, 1, 2, 3]
// Выпадающий список (строки)
Strings       = "foo";      // [foo, bar, baz]
// Выпадающий список с подписями [value:label, ...]
Size_label    = "S";        // [S:Small, M:Medium, L:Large]
// Чекбокс
Enable_thing  = true;
// Вектор ≤ 4 элементов → набор полей; можно с диапазоном
Vec3          = [12,34,46]; // [0:2:50]
```

**Группировка во вкладки:** блок-комментарий `/* [Имя вкладки] */` группирует параметры. Особые имена: `/* [Global] */` — параметр виден на всех вкладках; `/* [Hidden] */` — скрыт из GUI, но по-прежнему вычисляется (удобно для расчётных величин и зарезервированных под будущее параметров).

**Наборы параметров (headless):** значения можно хранить в JSON и подставлять без GUI — удобно для вариаций модели:
```bash
openscad-nightly --enable=customizer -o out.stl -p params.json -P setName model.scad
```

* **Локальные/расчетные переменные**: Пишутся в формате **camelCase** (например, `uHeight`, `boxOuterWidth`, `railThickness`).
* **Суффиксы единиц измерения**: Крайне рекомендуется указывать единицы измерения в названиях геометрических переменных, если они не очевидны (например, `_mm`, `XY`).
* **Модули**: Именуются с маленькой буквы в формате **camelCase** (например, `pallete()`, `canvasGrid()`, `orientedHexNutHole()`).
* **Внутренние/вспомогательные модули**: Для модулей, которые используются локально и не должны вызываться снаружи, используется префикс с подчёркиванием (например, `module _heart()`).

## 5. Предотвращение Z-fighting (мерцания соприкасающихся граней)

При вычитании (`difference`) или объединении объектов, чьи грани идеально совпадают, возникает графическая неопределенность (Z-fighting). 
* **Правило**: Всегда создавайте небольшой нахлёст (overlap) или используйте микро-сдвиг `fudge = 0.1` (или `0.01`).
* Вычитаемый объект должен быть заведомо больше/длиннее вырезаемой области и слегка смещен наружу.

**Пример без BOSL2:**
```openscad
fudge = 0.1;
difference() {
    cube([10, 10, 10]);
    translate([2, 2, -fudge]) // Сдвиг по Z вниз
        cube([6, 6, 10 + 2*fudge]); // Увеличенная высота
}
```

**Пример с BOSL2:**
Используйте встроенный аргумент `overlap` в трансформациях `attach()` или сдвигах:
```openscad
attach(BOT, TOP, overlap=0.01) cuboid([15, 10, 5]);
```

## 6. Использование библиотеки BOSL2

> 📌 Краткий справочник по правилам BOSL2 (ориентация/анкеринг, базовые примитивы, приёмы,
> чек-лист ошибок) вынесен в отдельный документ: **[BOSL2.md](./BOSL2.md)**.

Библиотека **BOSL2** (Belfry OpenSCAD Library v2) является основным стандартом для 3D-моделирования в проекте. Следует активно использовать её возможности для упрощения кода.

> **Примечание**: BOSL2 не является обязательной для простых деталей. Если модель тривиальна, не требует анкетринга/относительного позиционирования/параметризованных примитивов, а использование библиотеки не приносит реальной пользы — допустимо писать на «чистом» OpenSCAD (`translate()`/`rotate()`/`cube()`/`cylinder()`). Однако как только появляются множественные симметрии, повторяющиеся паттерны, скругления, сложные сечения или параметризуемые примитивы (трубы, призмоиды, петли) — переход на BOSL2 обязателен.
* **Относительное расположение деталей и трансформации (Anchoring, Attachments & Move/Flip/Copy)**: Крайне предпочтительно использовать относительное позиционирование и лаконичные трансформации из BOSL2 (такие как `attach()`, `move()`, `xmove()`, `ymove()`, `zmove()`, `xflip()`, `yflip()`, `mirror_copy()`, `xflip_copy()` и аналогичные) вместо стандартных и громоздких `translate()`, `mirror()`, `rotate()`. Это делает детали адаптивными, предотвращает поломку геометрии при изменении параметров, а код — чистым, плоским и цепочечным.
* **Использование сложных и готовых примитивов**: При проектировании деталей следует отдавать приоритет продвинутым готовым примитивам BOSL2 (таким как `tube()`, `prismoid()`, готовые петли/hinges и шарниры из `BOSL2/joiners.scad` или других библиотек соединений), избегая ручной сборки сложных геометрических тел из базовых кубов и цилиндров. **Важно**: Применяйте их только тогда, когда они действительно подходят под задачу, не усложняя геометрию избыточным функционалом там, где достаточно более простых базовых форм.
* **Анкеринг (`anchor`, `spin`, `orient`)**: Используйте привязку к осям и углам родительских и дочерних объектов для автоматического выравнивания.
* **Плавные переходы и траектории (`hull()`, `chain_hull()`)**: Для проектирования сглаженных деталей, органических переходов между сечениями разного диаметра и сложных гнутых траекторий активно применяйте стандартный `hull()` и библиотечный `chain_hull()` (последовательный hull между парами соседних дочерних элементов). Это избавляет от громоздких математических расчетов координат сопряжений. **Важно**: Помните, что операции `hull` крайне ресурсоемки при рендеринге (F6) — применяйте их с умом, взвешенно и только тогда, когда они действительно подходят под задачу, а более простые линейные/вращательные экструзии или встроенные скругления ребер (`rounding`, `chamfer`) не справляются.
* **Симметрия и паттерны**: Вместо циклов `for` применяйте `xflip_copy()`, `yflip_copy()`, `move_copies()` и `line_copies()`.
* **Чистые булевы операции**: Конструкции `diff()`, `union()` и теги (`tag("remove")`, `tag("keep")`) значительно облегчают управление вычитанием геометрии.

## 7. Модульность и вывод моделей (Инстанцирование)

Для облегчения отладки и подготовки деталей к 3D-печати:
1. Выносите логически независимые части в отдельные модули.
2. В самом низу файла оставляйте вызовы модулей. В процессе работы ненужные детали комментируются, чтобы в STL экспортировалась только целевая деталь.
3. При проектировании сложных сборок можно группировать вывод с помощью библиотек раскладки (например, `MCAD/layouts.scad` с модулем `list()`).

**Пример в конце файла:**
```openscad
// Выберите деталь для предпросмотра/экспорта:
// skoba(); 
ruchka(); 
```
